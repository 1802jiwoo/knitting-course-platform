# Project Status Rule

**상태 기호**
- [✓] 완료 / [⁓] 진행 중 / [ ] 미시작
- 기능 추가·변경 시 반드시 상태 반영

---

## Backend 작업 프로세스

1. 구현 완료 후 1차 코드 리뷰 보고 (구현 내용 요약, 변경된 API/로직 설명)
2. application.properties(yml) 확인 → active server URL 사용자에게 보고
3. 사용자 OK 이후 API 테스트 수행 (Postman 또는 테스트 코드 기준)
4. 테스트 결과 2차 보고 (요청/응답 결과, 성공/실패 명시)
5. 사용자 최종 OK 이후에만 적용 완료 처리

---

## 작업 선택 우선순위

1. 의존성 있는 작업
2. 핵심 기능
3. 영향도 높은 기능
4. 기타

---

## 멀티 작업 규칙

- 기본 1개 작업만 수행
- 반드시 필요한 의존 작업이 있을 경우에만 함께 수행 가능

## Project Status Rule (핵심)

* 작업을 시작된 경우 **진행 중인 작업 항목** 추가하고 향후 작업 항목에서 삭제
* 백엔드와 프론트중 하나만 진행된 경우 **진행 중인 작업** 항목에 넣음
* 백엔드와 프론트 모두 진행된 경우 **완료된 작업** 항목에 넣음
* 프론트만 작업된 경우 항목 옆에 **(프론트)** 라고 표시해야함
* 일부의 경우를 제외하곤 백앤드를 우선개발
* 모든 기능 변경 시 Project Status 업데이트 필수
* 상태: [✓], [⁓], [ ]
* 기능 추가 시 반드시 Status에 반영
